# 通用视频生产流程

这是本项目制作第 002 集及以后视频时的权威流程。第一集的具体经历和成品见 [第 001 集交付](../episodes/001-fitness-formulas-no-equals/README.md)。

## 0. 开工前

1. 执行 git pull。
2. 阅读 AGENTS.md、账号定位、剧集索引和上一集复盘。
3. 给新一集分配编号，复制 templates/episode。
4. 在 episodes/README.md 登记“策划中”。

## 1. 选题

选题来源可以是抖音、微博、公众号、新闻、公开报告或现实生活，但只借公共议题，不搬运成品。

每个选题先回答：

- 这件事和年轻打工人有什么关系？
- 它属于工作、生活金钱、关系，还是轻入口？
- 观众看完能获得什么：判断、解释、情绪松动或一个行动？
- 是否有争议事实需要核验？
- 是否能用固定 IP 表达成 5 至 8 个清楚画面？

## 2. 事实核验

在单集 01-research 中记录：

- 原始话题或链接。
- 至少一个可靠来源；重要或有争议结论尽量交叉核验。
- 已确认事实、仍有争议的部分和不可下的结论。
- 图片、视频、音乐和字体的授权状态。
- 是否使用 AI，以及发布时需要什么标识。

健康、法律、财务内容不得只依赖某个博主。优先使用官方机构、专业协会、系统综述或原始研究。

## 3. 写口播

先写“听起来顺”的口播，再写字幕，不要反过来。

推荐结构：

1. 开场：用自然场景引出问题。
2. 主体：每个画面只解释一个结论。
3. 收束：总结判断，连接账号的压力与内耗主线。

质量检查：

- 开场能否像正常聊天一样直接说出口？
- 每段是否可以独立配一张图？
- 是否把“可能”误写成“必然”？
- 是否重复解释同一件事？
- 是否有论文腔、公告腔或生硬转折？
- 结尾是否自然，而不是突然喊口号？

最终口播拆成独立代码块，一图一段，便于逐段复制到配音工具。

## 4. 设计分镜与图片

当前默认：

- 横版 16:9。
- 建议 1920×1080。
- 一段口播对应一张主图。
- 图片本身尽量不生成文字，避免乱码。
- 统一使用白色小电阻人 IP。
- 角色、重要动作和字幕避开边缘，给抖音裁切留安全区。

图片提示词至少说明：

- 参考图是角色身份参考。
- 必须保留的 IP 特征。
- 当前动作和情绪。
- 场景、构图、光线和留白。
- 禁止文字、Logo、水印、多余角色和错误肢体。

封面单独制作，不要直接拿信息最满的分镜图充当封面。封面只需要一个清楚动作和一块标题留白。

## 5. 在剪映中剪辑

建议顺序：

1. 新建 16:9 项目并按顺序导入所有图片。
2. 每段口播单独生成语音，不要先把整篇长稿一次性生成。
3. 先把第 1 段语音对齐第 1 张图，再依次处理后续段落。
4. 按语音长度调整图片时长，确认一一对应后再做效果。
5. 自动识别字幕并逐字校对。
6. 每张静态图使用轻微缩放，约从 100% 到 106%。
7. 转场统一用 0.2 至 0.3 秒叠化或淡化。
8. 加低音量背景音乐，确保人声始终清楚。
9. 设置独立封面。
10. 全片复核后导出。

最重要的顺序是：语音与图片对齐 → 时长 → 字幕 → 动效 → 转场 → 音乐 → 封面。不要先做满特效再返工对齐。

## 6. 导出前检查

- 七段或本集全部段落是否与图片准确对应？
- 字幕是否有错字、漏字、错误断句？
- 图片是否出现 AI 水印、乱码、多余手脚或角色漂移？
- 画面是否过于像 PPT？
- 人声是否被音乐盖住？
- 是否存在过度绝对或无依据的专业结论？
- 封面标题区是否清楚？
- 结尾是否至少停留约 0.5 至 1 秒？

## 7. 发布

发布材料保存在 06-publishing：

- 标题。
- 作品介绍。
- 3 至 5 个相关标签。
- 封面文字。
- AI 内容声明。
- 发布时间和状态。

当前新号的默认建议：

- 工作日优先 20:30 发布；中午 12:15 可作为测试时段。
- 发布后尽量在线 30 分钟，正常回复第一批评论。
- 初期关闭下载。
- 合拍、转发、评论和弹幕可以开启。
- 不添加无关的影视、演艺、小程序或活动标签。
- 一分钟左右的视频通常不需要章节要点。
- 使用平台提供的“内容由 AI 生成”标识；没有入口时在配文中说明画面由 AI 辅助创作。

平台规则会变化，发布当天仍需以抖音页面的最新提示为准。

## 8. 发布后复盘

在发布后 24 小时和 72 小时记录：

- 播放量。
- 2 秒或 5 秒留存。
- 完播率和平均观看时长。
- 点赞、评论、收藏、转发。
- 主页访问和新增关注。
- 高质量评论与观众新问题。

复盘重点不是只看播放量，而是判断：

- 开场有没有留住目标人群？
- 哪一段开始掉人？
- 观众把账号理解成了什么？
- 哪个问题值得做下一集？

## 9. 跨电脑交接

结束一轮工作前：

1. 更新本集 README 的状态和下一步。
2. 把最终文案、提示词和发布配文写入文件。
3. 更新 episodes/README.md。
4. 检查 git diff 和文档链接。
5. 提交并推送。

下一台电脑只需要 git pull，再让 Codex读取上述文件。
